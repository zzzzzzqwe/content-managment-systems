Лабораторная работа №5. Безопасность WordPress
Цель работы
Закрепить ключевые практики безопасности WordPress: управление ролями и паролями, обновления, базовое hardening (wp-config.php, права, отключение редактора), резервное копирование, мониторинг активности и поэтапная настройка All In One WP Security & Firewall (AIOS) для защиты от брутфорса, базового WAF и контроля прав.

Условие
Шаг 1. Подготовка среды
В локальной установке WordPress перейдите в админ-панель.
Убедитесь, что у вас есть доступ администратора.
Включите отладку в wp-config.php, установив define('WP_DEBUG', true);.
Шаг 2. Управление ролями и паролями
Создайте тестового пользователя c ролью Автор (для дальнейших проверок).
Проверьте, что у каждого администратора включены сложные пароли (8+ символов, буквы/цифры/символы).
Шаг 3. Обновления ядра, тем и плагинов
Проверьте наличие обновлений для WordPress, тем и плагинов.
Обновите всё до последних версий.
Настройте автоматические обновления для тем и плагинов.
Проверьте, что все обновления прошли успешно и сайт работает корректно.
Шаг 4. Базовое hardening
Запретите редактирование файлов из админки, добавив в wp-config.php: php define('DISALLOW_FILE_EDIT', true);
Настройте верные права на файлы и папки:
Папки: 755
Файлы: 644
Защитите wp-config.php, добавив в .htaccess: <files wp-config.php> order allow,deny deny from all </files>
Шаг 5. Установка и первичная настройка All In One WP Security & Firewall (AIOS)
Установите и активируйте плагин All In One WP Security & Firewall.
Перейдите в раздел плагина в админ-панели.
Настройте следующие параметры:
User Login:
Включите Login Lockdown. Рекомендуемые параметры для теста: Max Login Attempts: 5, Login Retry Time Period: 15 мин, Lockout Time: 30 мин.
Включите Force Logout (например, 24 ч), чтобы ограничить "вечные" сессии.
User Accounts:
Проверьте, нет ли пользователя с логином admin. Если есть — переименуйте через AIOS в безопасный логин.
User Registration:
Отключите автоподтверждение или включите ручное одобрение новых пользователей, если регистрация открыта.
Filesystem Security:
Запустите проверку File Permissions и примените рекомендованные исправления (не делайте мирозаписываемых прав).
Firewall:
Активируйте Basic Firewall (начните с базового уровня).
Включите защиту от Bad Query Strings, XSS, directory browsing.
Brute Force:
Включите Rename Login Page (измените URL входа с /wp-login.php на нестандартный, например /login-<slug>). Сохраните новый URL в менеджер паролей!
Scanner / Malware:
Настройте file change detection (уведомления на почту).
Backup:
В секции Database создайте резервную копию БД (храните вне веб-корня). Настройте расписание, если доступно.
Notifications:
Включите email-уведомления для важных событий (например, lockout, новый администратор, изменение файлов).
Шаг 6. Проверка защиты от брутфорса (на тестовом пользователе)
Выйдите из админки (или используйте приватное окно).
Перейдите на новый URL входа, попробуйте ввести неправильный пароль 5–6 раз.
Убедитесь, что сработал Lockdown (блокировка IP/пользователя).
Посмотрите запись о блокировке в WP Security → Dashboard / Logs и (по необходимости) разблокируйте тестовый IP.
Шаг 7. Восстановление из резервной копии
Удалите тестовую запись и одно произвольное изображение.
Восстановите БД из бэкапа (импорт SQL или через плагин).
Проверьте целостность данных (восстановлено ли удалённое изображение и тестовая запись).
